\section{Introduction}
% ==========================================================================

This project uses two datasets to investigate the relationship between stock market prices and residential property values. The first is the \textit{2000+ Assets (Stock \& FX) Historical Price} dataset from Kaggle, comprising 56.8 million rows across 3,762 individual CSV files totalling 3.12\,GB. At this scale, a standard single-node SQL pipeline took over five minutes to run on an Apple M4 MacBook Air (24\,GB RAM, macOS 26), as detailed in Section~2. The second is the S\&P/Case--Shiller U.S. National Home Price Index, a monthly composite tracking national residential property values. The two datasets differ in both structure and granularity: the stock data is spread across thousands of files at daily per-asset resolution, while the housing data is a single monthly time series. Combining them requires temporal alignment and per-ticker aggregation before any statistical analysis can be performed.

The aim of this project is to determine whether month-over-month changes in individual stock prices are correlated with changes in the national housing index. A Hadoop MapReduce pipeline was used to make this analysis practical at the given data scale, processing all assets without requiring the files to be merged in advance.

\section{Traditional Data Processing Baseline}
% ==========================================================================

\subsection{Architecture \& Setup}

\begin{itemize}
    \item \textbf{Hardware:} Apple M4 MacBook Air, 24\,GB RAM, macOS 26. Peak memory during SQL aggregation: $\sim$18\,GB.
    \item \textbf{Database Engine:} MySQL accessed via DataGrip.
    \item \textbf{Pipeline Design:} A sequential pipeline consisting of a single-threaded Python data preparation step followed by four SQL queries, each operating on the full dataset.
\end{itemize}

\subsection{Pipeline Implementation}

\textbf{Step 0: Data Preparation (Sequential I/O).}
MySQL cannot ingest a directory of separate files directly. A Python script was used to concatenate the 3,761 stock CSVs into a single 3.12\,GB file, adding a ticker column from each filename in the process.

\begin{lstlisting}[language=Python, columns=fullflexible]
import pandas as pd, glob, os

folder_path = "./data/1d"
all_files = glob.glob(os.path.join(folder_path, "*.csv"))
df_list = []

for file in all_files:
    df = pd.read_csv(file)
    df['Ticker'] = os.path.basename(file).split('.')[0]
    df_list.append(df)

big_df = pd.concat(df_list, ignore_index=True)
cols = ['Ticker', 'Date', 'Adj Close', 'Close', 'High', 'Low', 'Open', 'Volume']
big_df = big_df[cols]
big_df.to_csv('combined_stocks.csv', index=False)
\end{lstlisting}

\begin{lstlisting}[language=bash]

# 99.66s user  9.78s system  93% cpu  1:57.49 total
# Peak memory: ~4 GB
\end{lstlisting}

\textbf{Step 1: Temporal Aggregation.}
The 56.8\,M daily stock rows were grouped into monthly averages to match the granularity of the housing index.

\begin{lstlisting}[language=SQL]
CREATE TABLE monthly_stocks AS
SELECT
    Ticker,
    DATE_FORMAT(Date, '%Y-%m') AS YearMonth,
    AVG(Close)                 AS Monthly_Avg_Close
FROM stock_prices
GROUP BY Ticker, DATE_FORMAT(Date, '%Y-%m');
-- Runtime: 3m 04.85s  |  Output: 2.7M rows  |  Peak memory: ~18 GB
\end{lstlisting}

\textbf{Step 2: Data Integration (Inner Join).}
Monthly stock averages were joined with the Case--Shiller index on a shared \texttt{YYYY-MM} key, keeping only months present in both datasets (1987--2021).

\begin{lstlisting}[language=SQL]
CREATE TABLE joined_data AS
SELECT s.Ticker, s.YearMonth, s.Monthly_Avg_Close, h.Housing_Index
FROM monthly_stocks s
JOIN housing_data h ON s.YearMonth = DATE_FORMAT(h.Record_Date, '%Y-%m');
-- Runtime: 1.93s  |  Output: 1.58M rows
\end{lstlisting}

\textbf{Step 3: Feature Engineering (Window Functions).}
Month-over-month percentage change was computed for both series using \texttt{LAG} window functions. \texttt{NULLIF} prevents division by zero where historical close prices are \$0.00.

\begin{lstlisting}[language=SQL]
CREATE TABLE feature_engineered_data AS
SELECT Ticker, YearMonth,
    (Monthly_Avg_Close - LAG(Monthly_Avg_Close)
        OVER (PARTITION BY Ticker ORDER BY YearMonth))
    / NULLIF(LAG(Monthly_Avg_Close)
        OVER (PARTITION BY Ticker ORDER BY YearMonth), 0) AS Stock_Pct_Change,
    (Housing_Index - LAG(Housing_Index)
        OVER (PARTITION BY Ticker ORDER BY YearMonth))
    / NULLIF(LAG(Housing_Index)
        OVER (PARTITION BY Ticker ORDER BY YearMonth), 0) AS Housing_Pct_Change
FROM joined_data;
-- Runtime: 33.23s
\end{lstlisting}

\textbf{Step 4: Statistical Analysis (Pearson Correlation).}
The Pearson Correlation Coefficient ($r$) was computed per ticker. Tickers with fewer than 12 overlapping months were excluded.

\begin{lstlisting}[language=SQL]
CREATE TABLE final_correlation AS
SELECT Ticker, COUNT(*) AS Months_Correlated,
    (COUNT(*) * SUM(Stock_Pct_Change * Housing_Pct_Change)
        - SUM(Stock_Pct_Change) * SUM(Housing_Pct_Change))
    / NULLIF(SQRT(
        (COUNT(*) * SUM(Stock_Pct_Change * Stock_Pct_Change)
            - POW(SUM(Stock_Pct_Change), 2))
      * (COUNT(*) * SUM(Housing_Pct_Change * Housing_Pct_Change)
            - POW(SUM(Housing_Pct_Change), 2))
    ), 0) AS Pearson_Correlation
FROM feature_engineered_data
WHERE Stock_Pct_Change IS NOT NULL AND Housing_Pct_Change IS NOT NULL
GROUP BY Ticker HAVING COUNT(*) >= 12
ORDER BY Pearson_Correlation DESC;
-- Runtime: 1.57s  |  Output: 2,901 rows
\end{lstlisting}

\section{MapReduce Optimisation}
% ==========================================================================

\subsection{Bottleneck Identification \& Expectations}

Based on the timings in Section~2.3, Steps~0 and~1 are the bottlenecks, accounting for 89\% of total runtime ($\sim$5 minutes).

\textbf{Step~0 (Sequential I/O):} The need to pre-merge 3,761 files is a limitation specific to relational databases. Hadoop's \texttt{FileInputFormat} reads a directory of files as a single input, so this step is unnecessary and its 117-second cost is eliminated entirely.

\textbf{Step~1 (Single-Node Grouping):} Grouping 56.8 million rows on a single core is the main compute bottleneck. MapReduce assigns one file per mapper, so all 3,762 CSVs are processed in parallel. The shuffle phase then groups records by ticker before they reach the reducer, replacing the sequential \texttt{GROUP BY} scan.

\textbf{Expectation:} Removing Step~0 and parallelising Step~1 across two datanodes should reduce the combined ingestion and aggregation time by a factor of 3 to 5, depending on container availability and HDFS replication overhead.

\subsection{MapReduce Solution}

The implementation parallelises Steps~0--1 and combines Steps~2--4 into the reduce phase, avoiding any intermediate disk writes. The full source (\texttt{StockHousingCorrelation.java}) is on GitHub; the key logic is presented below.

\textbf{Key--Value Design.}
\begin{itemize}
  \item \textbf{Map output:} \texttt{key = ticker (Text)}, \texttt{value = "STOCK,YYYY-MM,closePrice (Text)"}
  \item \textbf{Reduce input:} all values for a given ticker, grouped and sorted by Hadoop's shuffle
  \item \textbf{Reduce output:} \texttt{key = ticker (Text)}, \texttt{value = "n\_months\textbackslash tpearson\_r (Text)"}
\end{itemize}

\textbf{Map Function.}

The mapper:
\begin{itemize}
  \item Processes one stock CSV per task (3,762 tasks in total).
  \item Reads the ticker from the filename (e.g.\ \texttt{AAPL.csv} $\to$ \texttt{AAPL}), since column~0 of each file is \texttt{Date}.
  \item Emits one key--value pair per valid row: \texttt{(ticker, "STOCK,YYYY-MM,closePrice")}.
\end{itemize}

\begin{lstlisting}[language=Java]
@Override
protected void setup(Context context) throws IOException, InterruptedException {
    String filename = ((FileSplit) context.getInputSplit()).getPath().getName();
    ticker = filename.replace(".csv", "");
}

@Override
public void map(LongWritable key, Text value, Context context)
        throws IOException, InterruptedException {
    String[] parts = value.toString().trim().split(",");
    if (parts[0].equalsIgnoreCase("Date")) return;
    if (parts.length >= 3 && !parts[2].trim().isEmpty()) {
        try {
            double closePrice = Double.parseDouble(parts[2].trim());
            String yearMonth = parts[0].trim().substring(0, 7);
            outKey.set(ticker);
            outValue.set("STOCK," + yearMonth + "," + closePrice);
            context.write(outKey, outValue);
        } catch (NumberFormatException e) { }
    }
}
\end{lstlisting}

\textbf{Reduce Function.}

For each ticker, the reducer:

\begin{enumerate}
  \item Computes monthly average close prices (equivalent to SQL Step~1).
  \item Sorts months chronologically.
  \item Computes month-over-month percentage change for both series (equivalent to SQL Step~3).
  \item Calculates the Pearson Correlation Coefficient (equivalent to SQL Step~4).
\end{enumerate}

The housing index is read directly from HDFS in \texttt{setup()} before any \texttt{reduce()} calls, which avoids routing it through the shuffle. Tickers with fewer than 12 overlapping months are dropped. The full reducer is on GitHub; the Pearson calculation is shown below.

\begin{lstlisting}[language=Java]
@Override
protected void setup(Context context) throws IOException, InterruptedException {
    FileSystem fs = FileSystem.get(context.getConfiguration());
    BufferedReader br = new BufferedReader(
        new InputStreamReader(fs.open(new Path("/input/correlation/CSUSHPINSA.csv"))));
    String line;
    while ((line = br.readLine()) != null) {
        String[] parts = line.split(",");
        if (parts.length == 2 && !parts[0].trim().equalsIgnoreCase("observation_date"))
            housingData.put(parts[0].trim().substring(0, 7),
                            Double.parseDouble(parts[1].trim()));
    }
    br.close();
}

// Pearson correlation over month-over-month pct changes
double numerator = (n * sumXY) - (sumX * sumY);
double denominatorSq =
    ((n * sumX2) - (sumX * sumX)) * ((n * sumY2) - (sumY * sumY));
if (denominatorSq > 0) {
    double correlation = numerator / Math.sqrt(denominatorSq);
    outValue.set(n + "\t" + String.format("%.6f", correlation));
    context.write(key, outValue);
}
\end{lstlisting}

The job was configured with a single reducer (\texttt{mapreduce.job.reduces=1}) and submitted to YARN (\texttt{mapreduce.framework.name=yarn}) for execution across two datanode containers.

\subsection{Critical Analysis}

The results largely match expectations, with an end-to-end speedup of \textbf{3.9$\times$}.

\begin{enumerate}

\item \textbf{I/O Efficiency:} The largest gain came from removing Step~0. Because \texttt{FileInputFormat} reads the 3,762-file directory in parallel, the 117-second Python merge was bypassed entirely, accounting for 35\% of the total baseline time.

\item \textbf{Scaling vs.\ Overhead:} Despite using two datanodes, the aggregation phase improved by roughly 3.4$\times$ rather than the theoretical 2$\times$ per-node figure. This is partly because the per-file parallelism across 3,762 mappers is more significant than the two-node split, but also because JVM container startup and shuffle network I/O add overhead that grows less significant at larger cluster sizes.

\item \textbf{Data Skew:} Some tickers had substantially more data than others. AAPL and ABT both have 421 months of records, while many recently listed assets have fewer than 50. This meant a small number of map tasks took noticeably longer than average, limiting the parallel efficiency.

\item \textbf{Correctness:} Output was cross-checked against the SQL results. Both pipelines use the same method and produce consistent rankings, with small numerical differences due to floating-point precision between Java \texttt{double} and MySQL decimal types. An earlier version passed housing data through the shuffle using a sentinel key (\texttt{000\_HOUSING}), but this produced no output in local mode because key ordering within a single JVM is not guaranteed. Loading the housing data in \texttt{setup()} resolved this.

\end{enumerate}

The MapReduce approach removed both the file I/O bottleneck and the single-node aggregation constraint that dominated the SQL pipeline runtime.

The MapReduce job produced Pearson Correlation Coefficients for \textbf{2,899 tickers}, cross-checked against the SQL baseline. Figure~\ref{fig:index_comparison} shows the two indices over time. Correlation values are low across all 2,899 assets. The highest value among tickers with at least 100 months of data is $r \approx 0.36$, and most fall below $|r| < 0.2$. This suggests that at monthly granularity, individual stock returns and housing price changes are largely independent, even though both are influenced by broader macroeconomic conditions. As Figure~\ref{fig:index_comparison} shows, the two markets responded differently to the 2008 financial crisis: housing peaked in 2006 and fell through 2012, while equity markets recovered more quickly. A similar divergence occurred after 2020. These periods reduce the month-over-month correlation precisely when the relationship between markets is most economically interesting. Sector-specific analysis, for example focusing on REITs or homebuilders, or using longer time lags, would likely produce stronger results. Note that FX pairs such as \texttt{GBPJPY-X} appear in the output since they were stored in the same input directory as the equities, and would need to be filtered out for any equity-specific analysis.

\section{Conclusion}
% ==========================================================================

This project built and compared two pipelines for computing Pearson correlations between individual stock returns and the U.S. housing index across 3,762 assets and 56.8 million records. The SQL baseline took 5\,m\,39\,s with a peak memory of $\sim$18\,GB, with 89\% of that time spent on file merging and aggregation. The Hadoop MapReduce implementation removed the merge step entirely and distributed the aggregation across 3,762 parallel map tasks, completing in 1\,m\,27\,s and reducing peak per-container memory to $\sim$2\,GB, a \textbf{3.9$\times$ speedup}.

The gap between the observed and theoretical speedup reflects JVM startup costs, shuffle I/O overhead, and skew from tickers with long data histories. These effects become proportionally smaller at larger cluster sizes.

On the analytical side, month-over-month stock returns show weak correlation with housing price changes for the vast majority of assets ($|r| < 0.2$ for most, maximum reliable $r \approx 0.36$). The two markets share broad long-run trends but diverge at major economic events, which suppresses short-term correlation. This is consistent with the cross-asset correlation literature at short timescales, and suggests that longer lags or sector-level groupings would be more informative.
